## 1. مبدأ التصميم الجوهري

**"المستخدم لا يضيع السياق أبداً"**

| نوع العملية | الأسلوب |
|------------|---------|
| عمليات سريعة (دفعة، موعد، صرف) | Drawer/Modal — بدون مغادرة الشاشة |
| عمليات كاملة (فاتورة جديدة، تقرير) | صفحة كاملة |
| التنقل بين الوحدات | Sidebar دائم ثابت لا يتغير |

**قيود التصميم:**
- الموظف غير التقني يرى فقط الوحدات التي يحتاجها (الوحدات الأخرى تختفي كلياً، لا تُعطَّل فقط)
- لا أكثر من نقرتين للوصول لأي عملية استقبال شائعة
- بيانات المريض الحرجة (حساسية، فصيلة دم) ثابتة دائماً وواضحة في الشاشة السريرية

## 2. Information Architecture

```
النظام
├── المستوى 1: 13 وحدة رئيسية
│   ├── لوحة التحكم (عملياتية)
│   ├── لوحة الإدارة (Executive) ← مدير فقط
│   ├── الاستقبال
│   ├── المرضى
│   ├── العيادة (الطبيب)
│   ├── المواعيد
│   ├── الخزينة ← مُعاد هيكلتها بالكامل (V2)
│   ├── المخزون ← + تقارير الاستهلاك (V2)
│   ├── المشتريات ← + حساب مالي للمورد (V2)
│   ├── المصروفات / مراكز التكلفة ← مستقلة (V2)
│   ├── التأمين ★ Core Light (V2 — كانت Optional)
│   ├── الجهات الخارجية ★ جديدة (V2)
│   └── التقارير
│
├── المستوى 2: شاشات فرعية (67 شاشة)
│
├── المستوى 3: Tabs / تفاصيل داخل كل شاشة
│   مثال: ملف المريض → [البيانات | الزيارات | مكتبة الوسائط | الحساب المالي | خطط العلاج]
│   مثال: ملف المورد → [البيانات | المشتريات | الحساب المالي]
│   مثال: ملف الطبيب → [البيانات | الجدول | الحساب المالي]
│
└── المستوى 4: Modals / Drawers
    مثال: استلام دفعة = Drawer من اليمين
    مثال: حركة عكسية = S65 (بلا موافقة، مباشر)
```

**تغيير جوهري V2 عن V1:**
- التأمين لم تعد تظهر بخط رمادي/معطّل — أصبحت Core Module ظاهرة دائماً
- Lab و Radiology فقط بقيا Optional/Pluggable (معطّلين افتراضياً)
- "الفروع" محذوفة كلياً — لا Branch Selector في أي شاشة

```
    │
    ▼

                 رفض  → الفاتورة تعود لحالتها + سبب الرفض


الحالة ب — تصحيح دفعة خاطئة (لا يحتاج Approval — Reverse فقط):
────────────────────────────────────────────────────────────────
محاسب يكتشف خطأ في مبلغ/خزينة دفعة
    │
    ▼
يفتح S65 — حركة عكسية
    بحث عن الحركة الأصلية برقم مرجعي
    سبب العكس (إلزامي)
    ☐ إنشاء حركة صحيحة بديلة مباشرة
    │
    ▼
النظام لا "يعدّل" الحركة الأصلية أبداً
بل يُنشئ: حركة أصلية + حركة عكسية + حركة صحيحة (3 سجلات)
تنبيه معلوماتي للمدير (إشعار فقط — لا قرار مطلوب)
```

### Flow C: استلام/دفع — نموذج خزينة ديناميكي (V2)

```
[Pills اختيار نوع الحركة]:
استلام من مريض | دفع لمريض | استلام من مورد | دفع لمورد
دفع طبيب | دفع فني | دفع موظف | استلام عام | صرف عام

الحقول تتغيّر ديناميكياً حسب النوع:
  "استلام من مريض"  → حقل: اختيار مريض + ربط بفاتورة
  "دفع لمورد"       → حقل: اختيار مورد + ربط بفاتورة شراء

  "صرف عام"         → حقل: مركز التكلفة (إلزامي)

الحقول المشتركة: المبلغ | طريقة الدفع | الخزينة | ملاحظة
```

## 11. Mobile Responsive Strategy

**القرار: Desktop-First مع Tablet Support كامل، Mobile محدود**

| الجهاز | الشاشات المدعومة | ملاحظات |
|--------|-----------------|---------|
| Desktop (xl+) | كل الشاشات | Sidebar مفتوح 260px |
| Tablet landscape (lg) | كل الشاشات | Sidebar قابل للطي |
| Tablet portrait (md) | كل الشاشات | Sidebar كـ Drawer (Hamburger) |
| Mobile (sm) | محدود جداً | رسالة "استخدم تابلت أو كمبيوتر" |

**سلوك Sidebar:**
- `xl+`: ثابت مفتوح (260px)
- `lg`: قابل للطي (Icons only 64px)
- `md-`: Drawer منبثق (☰ Hamburger)

**سلوك الجداول على الموبايل:**
- تحويل إلى Card List (كل صف = بطاقة عمودية)
- مثال: قائمة الانتظار = بطاقات مع [استدعاء]

**Dental Chart على Tablet:**
- نفس SVG + Touch Target 44px+ لكل منطقة سن

**ما لا يُدعم على الموبايل:**
- شاشات المخزون والمشتريات المعقّدة
- الإعدادات المتقدمة

### قواعد SignalR (Frontend)

```typescript
// الاتصال
import * as signalR from '@microsoft/signalr'

const connection = new signalR.HubConnectionBuilder()
  .withUrl('/hubs/dental-erp')
  .withAutomaticReconnect()
  .build()

// استقبال أحداث الطابور
connection.on('QueueUpdated', (queueData) => {
  queryClient.invalidateQueries({ queryKey: ['queue'] })
})

// مؤشر الاتصال الحي
<RealtimeIndicator connected={connection.state === 'Connected'} />
```

### قواعد TanStack Query

```typescript
// Server State: TanStack Query v5
// Client State: Zustand

// مثال Query للفواتير
const { data: invoices } = useQuery({
  queryKey: ['invoices', filters],
  queryFn: () => api.get('/api/v1/invoices', { params: filters }),
  staleTime: 30_000,

// مثال Mutation لاستلام دفعة
const receivePayment = useMutation({
  mutationFn: (data) => api.post('/api/v1/payments', data),
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ['invoices'] })
    queryClient.invalidateQueries({ queryKey: ['vault'] })
    toast.success('تم استلام الدفعة بنجاح')
  },
})
```

## ⚠️ نقاط تحتاج توضيح

1. **S06 شاشة الاستدعاء (Display Screen):** الوثائق تذكرها لكن لم تُفصّل — هل هي شاشة عرض خارجية منفصلة (TV في الانتظار)؟ أم Popup للمريض؟ يحتاج تحديد نمط العرض ودورة الحياة.

3. **الأرقام الهندية vs اللاتينية في التقارير:** الوثيقة توصي بلاتينية (123) — هل هذا قرار نهائي أم يُترك للمستخدم كإعداد؟

4. **S31b الرواتب:** مذكورة في Screen Inventory لكن لم تُفصّل — هل هي حساب رواتب كامل أم مجرد "دفع موظف" من الخزينة؟ يؤثر على ERD (هل نحتاج payroll module مستقل؟).

5. **S17 "إرسال للخزينة":** هل هي شاشة كاملة أم Modal تأكيد بسيط؟ وهل يمكن للطبيب تعديل السعر النهائي في هذه الخطوة؟
